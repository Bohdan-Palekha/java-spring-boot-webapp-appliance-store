1. Spring Data JPA — Every entity has a dedicated JpaRepository. Custom @Query JPQL methods are used across OrdersRepository, EmployeeRepository, ManufacturerRepository, and UserRepository for complex queries like findByClientIdWithFullDetails() (JOIN FETCH to avoid N+1), isLastAdmin(), and searchUsers(). ApplianceRepository additionally extends JpaSpecificationExecutor for dynamic search.
2. Spring Security — SecurityConfig.java is the central hub. It configures STATELESS session policy (JWT-driven), granular URL rules (/admin/** → ADMIN/EMPLOYEE, /orders/** → CLIENT only), form login with custom success/failure handlers, OAuth2 (Google OIDC), and the JWT filter chain. CustomUserDetailsService handles DB-backed auth, assigning ROLE_ADMIN+ROLE_EMPLOYEE or ROLE_CLIENT based on the User subtype and department field.
3. DTOs — Five form DTOs (ApplianceFormDTO, RegisterFormDTO, OrderFormDTO, ManufacturerFormDTO, ClientEditDTO) act as validation boundaries, keeping entities clean and decoupling the HTTP layer from the domain.
4. SQL Scripts — data.sql seeds manufacturers, appliances, 4 users (with BCrypt hashes), and sample orders. Activated automatically via spring.sql.init.mode: always in dev; disabled (never) in prod.
5. Internationalisation — I18nConfig.java wires ReloadableResourceBundleMessageSource to SessionLocaleResolver with a ?lang=uk interceptor. Both messages.properties (English) and messages_uk.properties (Ukrainian) cover 100+ keys including all validation errors. The LocalValidatorFactoryBean is tied to the same MessageSource so validation errors are also localised.
6. Validation — @Valid + BindingResult in every controller, rich Jakarta constraints on DTOs, a custom @UniqueEmail constraint (hits the DB), and Bootstrap is-invalid field highlighting in Thymeleaf templates.
7. Exception Handling — Abstract AppException base class, 9 specific subclasses, and GlobalExceptionHandler (@ControllerAdvice) mapping each to the correct HTTP status and locale-aware error page. Stack traces are suppressed in all environments.
8. Logging — AOP LoggingAspect instruments every service method (@Around) for timing (WARN if >500ms), and every auth attempt (@Before/@AfterReturning/@AfterThrowing). Debug through Error levels are all used with meaningful prefixes ([SVC], [AUTH], [JWT], [SECURITY]).
9. Database Configuration — H2 in-memory (dev/test, create-drop, MODE=PostgreSQL) and PostgreSQL (prod, ddl-auto: validate, HikariCP pooling). Sensitive credentials are externalised via environment variables in prod.
10. Unit Testing — Service tests (Mockito) and controller tests (@WebMvcTest) cover all major classes. Service tests verify business rules (last-admin guard, order state guard, lockout logic). Controller tests verify HTTP status codes, redirects, and security role enforcement.
11. Order Management — Full lifecycle: client places order → pending state (approved=null) → employee approves/rejects → IllegalOrderStateException prevents double-processing. setApprovalStatus() records which employee made the decision.
12. JWT is fully implemented with 15-min access tokens and 7-day refresh tokens stored in the DB. Logout revokes the refresh token and deletes the JWT cookie.
13. BCrypt cost-12 is used everywhere passwords are stored.
